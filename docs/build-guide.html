01前端 · Fluttervelvet-flutter / main

环境准备

# 检查环境
flutter --version    # 必须 3.5 及以上
flutter doctor       # 全绿后再继续

# 拉代码
git clone https://gitlab.qmpexevcqy.work/flutter_apps/velvet-flutter.git
cd velvet-flutter
git checkout main
flutter pub get

Android APK · 测试 / 内部分发

分架构包（推荐，每个 APK 体积小约 30MB）：

flutter clean
flutter pub get
flutter build apk --release --split-per-abi

# 产物路径：
#   build/app/outputs/flutter-apk/app-armeabi-v7a-release.apk
#   build/app/outputs/flutter-apk/app-arm64-v8a-release.apk      ← 主流机型
#   build/app/outputs/flutter-apk/app-x86_64-release.apk

单一通用包（兼容所有架构，体积约 80MB+）：

flutter build apk --release
# 产物：build/app/outputs/flutter-apk/app-release.apk

Android AAB · 上架 Google Play

flutter build appbundle --release
# 产物：build/app/outputs/bundle/release/app-release.aab

iOS IPA · 必须 macOS + Xcode 环境

flutter build ipa --release
# 产物：build/ios/ipa/*.ipa

# 上传 App Store Connect：
xcrun altool --upload-app -f build/ios/ipa/velvet.ipa \
  -u <apple-id> -p <app-specific-password>

需要 Apple Developer 账号 + 已配置好的描述文件 (Provisioning Profile) 与发布证书 (Distribution Certificate)，由我方提供。

Web · H5 部署

flutter build web --release --web-renderer canvaskit
# 产物：build/web/  →  整个目录上传静态服务器

02后端 · Spring BootJDK 21 · Maven 3.9+ · 容器化部署

方式一 · 纯 JAR（裸机部署）

cd velvet-backend
mvn clean package -DskipTests -B

# 产物：target/backend-0.1.0-SNAPSHOT.jar

# 启动：
java -jar target/backend-0.1.0-SNAPSHOT.jar \
  --spring.profiles.active=prod

方式二 · Docker 镜像（推荐）

仓库根目录已附多阶段 Dockerfile（Maven build + JRE 21 runtime + 非 root 用户）。

cd velvet-backend

# 构建镜像
docker build \
  -t velvet-backend:0.1.0 \
  -t velvet-backend:latest .

# 本地测试运行
docker run -d --name velvet-backend -p 8080:8080 \
  --env-file .env.prod \
  velvet-backend:latest

# 推送至镜像仓库（registry 地址由我方提供）
docker tag velvet-backend:latest \
  <registry>/velvet-backend:0.1.0
docker push <registry>/velvet-backend:0.1.0

方式三 · 跳过测试快速打包

mvn clean package -DskipTests -Dmaven.test.skip=true -B -T 4
# -T 4 = 4 线程并行编译，提速明显
运行时配置
容器暴露端口 8080，健康检查路径 /api/v1/health，启动宽限期 60 秒。Kubernetes readinessProbe 与 livenessProbe 沿用此路径即可。
环境变量 / 配置文件
生产配置走 --spring.profiles.active=prod，对应 application-prod.yml。所有敏感配置（数据库 / Redis / OSS / 第三方 API key）通过环境变量注入，.env.prod 模板由我方在交接时单独提供，不在仓库中。

03Android 签名上架前必做

仓库当前 android/app/build.gradle.kts 配置使用 debug key 签 release（开发期临时方案）。正式发版必须替换为 release keystore，否则 Google Play 拒收。

步骤一 · 生成 keystore

# 提包组本地或我方专机生成均可，但密钥文件不进 git
keytool -genkey -v \
  -keystore velvet-release.jks \
  -keyalg RSA -keysize 2048 -validity 10000 \
  -alias velvet

步骤二 · 创建 android/key.properties

.gitignore 已覆盖此文件，不会进库。

storePassword=<keystore 密码>
keyPassword=<key 密码>
keyAlias=velvet
storeFile=<velvet-release.jks 绝对路径>

步骤三 · 修改 build.gradle.kts

将 android/app/build.gradle.kts 中 release block 内的 signingConfig = signingConfigs.getByName("debug") 改为读取 key.properties 的 release config。具体改法依官方文档 Flutter · Signing the app。可由我方代改后再交付，二选一即可。

关键风险
keystore 文件丢失或密码遗忘后，App ID com.velvet.velvet 在 Google Play 将永久无法更新。请至少备份至 2 处独立加密存储（U 盘 + 加密云盘 / 1Password 等密钥管理服务）。

04交付物清单每次打包请按此回交

Android 测试包
  • app-arm64-v8a-release.apk（主流机型）
  • app-armeabi-v7a-release.apk（兼容 32 位老机）
  • versionName + versionCode（写在文件名或交接表格）
  • SHA-256 校验值（apksigner verify --print-certs）
Android 上架包
  • app-release.aab
  • 同时附 mapping.txt（混淆映射，build/app/outputs/mapping/release/）
  • 本次打包使用的 keystore 指纹（SHA-1 + SHA-256）
iOS 上架包
  • velvet.ipa
  • App Store Connect 上传成功截图或 Build 编号
  • 本次使用的 Provisioning Profile 名称
后端镜像
  • 镜像仓库完整地址 + 标签（如 registry.example.com/velvet-backend:20260427）
  • 镜像 digest（docker inspect 输出）
  • 本次构建对应的 git commit SHA（短 7 位即可）
每次构建前请确认
1. git pull origin main 拉到最新 · 2. 截图保留 commit SHA · 3. 构建过程出现的 warning / error 全部归集后随交付物一起回报，不要静默忽略。
